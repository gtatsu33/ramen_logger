1. アプリケーション概要
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ラーメン食べ歩きの記録を管理するシングルユーザー向けWebアプリ。

主な機能:
  - ラーメン店舗の登録（地図クリックまたは手入力）
  - 食べ歩き記録の登録（日付・ラーメン名・点数・コメント・写真）
  - 登録済み記録の統計表示（年別・月別・店舗別）
  - 写真グリッド表示とモーダル詳細表示
  - 登録済み店舗の地図表示

認証: 不要（シングルユーザー前提、アクセス制御はSupabase RLS側で行う）


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
2. データモデル
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

データストアは Supabase (PostgreSQL) を使用する。

────────────────────────────────────────
2-1. stores テーブル（店舗）
────────────────────────────────────────

  カラム名      型              制約             説明
  ----------   ----------      -----------      --------------------------------
  id           integer         PK, auto         主キー（自動採番）
  name         text            NOT NULL         店舗名
  latitude     float8          nullable         緯度（地図未選択時はNULL）
  longitude    float8          nullable         経度（地図未選択時はNULL）
  created_at   timestamptz     NOT NULL         登録日時（UTC）

────────────────────────────────────────
2-2. entries テーブル（食べ歩き記録）
────────────────────────────────────────

  カラム名      型              制約             説明
  ----------   ----------      -----------      --------------------------------
  id           integer         PK, auto         主キー（自動採番）
  date         text            NOT NULL         訪問日（YYYY-MM-DD 形式）
  store_id     integer         FK → stores.id   店舗ID
  ramen_name   text            NOT NULL         ラーメンの名前
  score        integer         NOT NULL         点数（1〜5 の整数）
  comment      text            nullable         コメント（空文字可）
  photo_path   text            nullable         写真のURL（512×512 正方形）
  created_at   timestamptz     NOT NULL         登録日時（UTC）

  リレーション:
    entries.store_id → stores.id（1店舗に複数記録）

────────────────────────────────────────
2-3. ファイルストレージ（Supabase Storage）
────────────────────────────────────────

  バケット名: ramen-photos
  公開設定: public（URLを知っていれば誰でも閲覧可能）

  ファイル命名規則:
    {uuid_hex}.jpg   例: a3f9c1d2...jpg

  photo_path カラムには Supabase Storage の公開URLを格納する。

3. 外部サービス
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

────────────────────────────────────────
3-1. Supabase
────────────────────────────────────────

  用途: データベース（PostgreSQL）・ファイルストレージ
  必要な設定値:
    SUPABASE_URL    : プロジェクトURL（例: https://xxxx.supabase.co）
    SUPABASE_KEY    : anon key（クライアントから使用可能な公開キー）

  使用するAPI:
    - stores テーブルの SELECT（全件・name昇順）
    - stores テーブルの INSERT
    - entries テーブルの SELECT（stores を JOIN、date降順）
    - entries テーブルの INSERT
    - Storage バケット "ramen-photos" へのファイル UPLOAD
    - Storage バケット "ramen-photos" の公開URL取得

────────────────────────────────────────
3-2. Google Maps JavaScript API
────────────────────────────────────────

  用途: 店舗位置の選択・店舗一覧の地図表示
  必要な設定値:
    GOOGLE_MAPS_API_KEY : Google Maps JavaScript API キー

  使用するライブラリ:
    - Maps JavaScript API（地図表示・マーカー）
    - Places API（nearbySearch：地図クリック周辺の店舗名取得）
    - Geometry ライブラリ（computeDistanceBetween：距離計算）

  APIキーの公開に関する注意:
    フロントエンドから直接利用するためキーはHTMLに含まれる。
    Google Cloud Console でリファラー制限（許可するドメインを限定）を設定すること。

4. 画像処理仕様
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

アップロード時に以下の処理を行う。処理はクライアントサイドでもサーバーサイドでも可。

────────────────────────────────────────
4-1. フルサイズ画像
────────────────────────────────────────

  1. EXIF 情報に基づき画像を正しい向きに回転（auto-orient）
  2. カラーモードが RGB でない場合（RGBA, グレースケール等）は RGB に変換
  3. 短辺を基準に中央正方形クロップ
     - 元サイズ (W, H) の場合:
       crop_size = min(W, H)
       left = (W - crop_size) / 2
       top  = (H - crop_size) / 2
       → 切り出し範囲: (left, top, left+crop_size, top+crop_size)
  4. 512×512px にリサイズ
  5. JPEG 形式に変換、品質 80

────────────────────────────────────────
4-2. アップロード手順
────────────────────────────────────────

  1. 画像を処理（4-1）しバイト列を得る
  2. {uuid_hex}.jpg として Supabase Storage にアップロード
  3. ファイルの公開URLを取得し、entries.photo_path に格納

  ※ サムネイルは生成しない。グリッド表示時はブラウザの描画エンジンが縮小する。

5. 画面構成と UI 遷移
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

アプリは基本的に単一ページ（SPA相当）。
画面は上から「記録入力エリア」→「統計エリア」の縦並びで構成される。

────────────────────────────────────────────────────────────────────────────
                         [ 画面全体のレイアウト ]
────────────────────────────────────────────────────────────────────────────

  ┌─────────────────────────────────────────────────────────┐
  │  🍜 ラーメンロガー  （タイトル）                           │
  │  訪問日、写真、お店、ラーメン名、点数、コメントを登録して    │
  │  統計を確認できます。 （サブキャプション）                  │
  ├─────────────────────────────────────────────────────────┤
  │  [記録入力エリア]                                        │
  │    店舗選択ドロップダウン                                  │
  │    ├─ 既存店舗選択時: 選択完了（地図なし）                  │
  │    └─ 「新しいお店を登録」選択時:                          │
  │           手入力テキストボックス（お店の名前）              │
  │           Google Maps（クリックで位置選択）                │
  │    記録フォーム                                           │
  │      訪問日 / ラーメン名 / 点数スライダー / コメント / 写真  │
  │      [保存する] ボタン                                    │
  ├─────────────────────────────────────────────────────────┤
  │  [統計エリア]  ─────────────────────                     │
  │  タブ: [年・月別]  [店舗別]                               │
  │  （タブ詳細は後述）                                        │
  └─────────────────────────────────────────────────────────┘

────────────────────────────────────────
5-1. 記録入力エリア
────────────────────────────────────────

【店舗選択ドロップダウン】
  - 先頭に「新しいお店を登録」という固定選択肢を表示
  - 以降に登録済み店舗を「店舗名 (緯度,経度)」形式で name 昇順に表示
  - 初期選択: 「新しいお店を登録」

【既存店舗を選択した場合】
  - 地図・手入力テキストボックスは非表示
  - 選択した店舗の name / latitude / longitude を内部で保持

【「新しいお店を登録」を選択した場合】
  以下の2つの入力手段を提供する（どちらか一方、または両方を使用可）:

  [A] 手入力テキストボックス（「お店の名前（手入力）」）
    - 自由記述
    - 最後に操作した側（手入力 or 地図クリック）の値が登録に使われる

  [B] Google Maps による位置選択
    - 表示サイズ: 高さ 400px、幅 100%
    - 初期表示: 現在地（ブラウザの Geolocation API で取得）
      取得できない場合は東京駅付近（緯度 35.681236, 経度 139.767125）
    - 地図上をクリックすると:
      1. クリック地点にマーカーを立てる
      2. Places API の nearbySearch で周辺の「ラーメン」キーワード店を検索
         → 最近傍店が 20m 以内なら店舗名を自動取得
      3. ラーメン店が見つからない場合は「restaurant」タイプで再検索
         → 最近傍店が 50m 以内なら店舗名を自動取得
      4. 店舗名が取得できた場合: 地図上部インジョインフォバーに「📍 {店舗名}」を表示
      5. 店舗名が取得できなかった場合: 「緯度 XX.XXXXXX, 経度 XX.XXXXXX（店舗名未検出）」を表示
    - クリック後: 緯度・経度・店舗名を内部に保持
    - 位置が選択済みであれば「選択された位置: 緯度 XX.XXXXXX, 経度 XX.XXXXXX」の
      確認メッセージを地図下に表示

  【店舗名の決定ロジック（新規登録時）】
    - 「手入力」と「地図クリック」の、最後に操作した方の店舗名を使用する
    - 地図クリック後に手入力を変更 → 手入力の名前を使用
    - 手入力後に地図をクリック → 地図で取得した名前を使用

【記録フォーム】
  フォーム内の項目:
    訪問日    : 日付選択（デフォルト: 本日）
    ラーメン名: テキスト入力（必須）
    点数     : 1〜5 の整数スライダー（デフォルト: 4）
    コメント  : テキストエリア（任意）
    写真     : 画像ファイル選択（PNG / JPG / JPEG）（必須）

  [保存する] ボタン押下時のバリデーション:
    1. ラーメン名が空 → エラー表示「ラーメンの名前を入力してください。」
    2. 新規店舗で店舗名が空 → エラー表示「新しいお店の名前を入力してください。」
    3. 写真が未選択 → 「写真未添付確認ダイアログ」を表示（後述）
    4. 全て OK かつ新規店舗 → 「新規店舗登録確認ダイアログ」を表示（後述）
    5. 全て OK かつ既存店舗 → 即座に保存処理を実行

  保存処理（既存店舗）:
    1. 写真をアップロード（4章の処理）
    2. entries テーブルに INSERT
    3. 保存完了メッセージを表示
    4. フォームをリセット・画面を更新

────────────────────────────────────────
5-2. ダイアログ / モーダル
────────────────────────────────────────

【写真未添付確認ダイアログ】
  タイトル: 「確認」
  本文: 「写真がアップロードされていません。」
  ボタン: [キャンセル]（ダイアログを閉じ、フォームに戻る）
  ※ 現在の仕様では写真なしでの保存は「しない」（キャンセルのみ提供）

【新規店舗登録確認ダイアログ】
  タイトル: 「登録確認」
  本文: 「店舗名「{店舗名}」で登録します。よろしいですか？」
  ボタン:
    [はい]（左）:
      1. stores テーブルに店舗を INSERT
      2. 写真をアップロード
      3. entries テーブルに記録を INSERT
      4. フォームをリセット・画面を更新
    [キャンセル]（右）:
      ダイアログを閉じ、フォームに戻る

【ラーメン詳細モーダル】
  写真グリッドの画像をクリックした際に表示。
  閉じるボタン（×）は非表示とし、代わりにフッターの「閉じる」ボタンのみで閉じる。

  表示内容（縦並び）:
    写真（フルサイズ、横幅いっぱい）
    🍜 {ラーメン名}
    📅 {訪問日}        ⭐ {点数} 点
    📍 {店舗名}
    💬 {コメント}（コメントがある場合のみ）
    [閉じる] ボタン

────────────────────────────────────────
5-3. 統計エリア（タブ切り替え）
────────────────────────────────────────

タブは2つ: 「年・月別」 と 「店舗別」

━ タブ1: 年・月別 ━━━━━━━━━━━━━━━━━━━━━━━━

  【年別来店回数テーブル】
    年     | 回数
    -------+------
    2025   |  12
    2024   |   8
    ...（year 昇順）

  【月別写真グリッド】
    年選択ドロップダウン（降順）
    選択した年の全記録を月ごとにグループ化して表示。
    月は降順（新しい月が上）。

    各月ブロックの構成:
      「{月}月 {件数}件」 （例: 「3月 4件」）
      写真グリッド（後述 5-4）

    ※ photo_path が NULL の記録はグリッドに表示しない

━ タブ2: 店舗別 ━━━━━━━━━━━━━━━━━━━━━━━━━

  【店舗別・年別記録】
    年選択ドロップダウン（降順）
    選択した年のデータを訪問回数の多い順に店舗を並べて表示。

    各店舗ブロックの構成:
      「{店舗名} {件数}件」
      写真グリッド（後述 5-4）
        ※ 当該店舗・当該年の記録のみ・photo_path があるもののみ・date 降順

  【登録済みのお店】
    登録済み店舗の一覧を表示。
    緯度・経度が設定されている店舗が1件以上ある場合: Google Maps に全マーカーを表示
    全店舗が緯度・経度なしの場合: 店舗名のテーブルを表示

    地図の設定:
      中心座標: 登録済み店舗リストの先頭の緯度・経度
      ズームレベル: 12
      各店舗のマーカーをクリックすると、吹き出しに店舗名を表示

────────────────────────────────────────
5-4. 写真グリッド
────────────────────────────────────────

  表示列数: 4列（固定）
  各セル:
    - photo_path があれば画像を表示（ブラウザがセルサイズに縮小描画）
    - なければプレースホルダー（グレーの正方形＋📷アイコン）を表示
  各セルの縦横比: 1:1（正方形）
  セル間のギャップ: 2px
  モバイルでも4列を維持する（ブラウザの自動折り返しを抑制）

  クリック動作:
    - 画像クリック → 「ラーメン詳細モーダル」（5-2）を表示
    - クリック検出は画像単位（座標は問わない）

6. 画面遷移・状態管理
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

アプリはページ遷移なし（単一ページ）。すべてモーダルダイアログで完結する。

状態として保持すべき情報:
  - 選択中の店舗（既存店舗IDまたは「新規」フラグ）
  - 地図で選択した緯度・経度・店舗名
  - 手入力の店舗名
  - 最後に操作した入力手段（「manual」or「map」）
  - フォームの各入力値（訪問日・ラーメン名・点数・コメント・写真）
  - 新規店舗保存前の「確認待ちデータ」（確認ダイアログに渡すデータセット）
  - 詳細モーダルに表示する対象エントリ
  - 各ダイアログの表示/非表示フラグ

保存成功後の動作:
  - フォームの全入力値をリセット
  - entries の一覧を再取得（統計・グリッドを最新状態に更新）
  - 地図の選択状態をリセット

7. APIアクセスパターン
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

以下の操作はすべてクライアントから Supabase REST API 経由で実行する。
（Supabase JS クライアント、Python クライアント、直接 HTTP のいずれでも可）

  操作                     テーブル     メソッド   条件・備考
  ──────────────────────   ─────────   ────────   ──────────────────────────────
  店舗一覧取得             stores       SELECT     全件・name 昇順
  店舗登録                 stores       INSERT     name, latitude, longitude, created_at
  記録一覧取得             entries      SELECT     stores を JOIN・date 降順
  記録登録                 entries      INSERT     全カラム
  写真アップロード         （Storage）  UPLOAD     バケット: ramen-photos
  公開URL取得             （Storage）  GET URL    アップロード後に取得

  ※ UPDATE / DELETE は現在の仕様では不使用

付録A. 既存写真データの一括変換バッチ処理
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

本アプリ外で一度だけ手動実行する移行スクリプト。
アプリ本体のコードとは独立しており、実行後は廃棄してよい。

────────────────────────────────────────
目的
────────────────────────────────────────

  既存の entries レコードの photo_path に格納されている画像は
  「アスペクト比保持・最長辺1280px」の非正方形画像である。
  新仕様の512×512正方形に統一するため、一括変換する。

────────────────────────────────────────
実行タイミング
────────────────────────────────────────

  新アプリのデプロイ前後どちらでも可。
  （新アプリが CSS で object-fit: cover を使えば
    変換前でも見た目は正方形に揃うため、緊急度は低い）

────────────────────────────────────────
処理手順
────────────────────────────────────────

  1. entries テーブルから photo_path を全件取得
  2. 各 photo_path の画像を Supabase Storage からダウンロード
  3. 4-1章と同じ変換処理を適用（正方形クロップ → 512×512 → JPEG品質80）
  4. 同じファイル名で Supabase Storage に上書きアップロード
     → ファイル名・URLが変わらないため entries テーブルの更新は不要

────────────────────────────────────────
注意事項
────────────────────────────────────────

  - 上書きアップロード前に Storage のバックアップを取ることを推奨
  - thumbnail_path カラムのデータはそのまま残してよい（新アプリは参照しない）
  - バッチ実行中もアプリは通常通り稼働可能（書き込み競合のリスクは低い）
